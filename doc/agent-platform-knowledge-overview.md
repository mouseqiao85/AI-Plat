# 数字员工仿真平台功能与架构说明

## 1. 平台定位

[[数字员工仿真平台]] 是一个面向企业、产品团队、开发团队和垂直行业场景的 [[Agent平台]]。平台的核心目标是把大模型能力、可复用 [[Skill]]、知识图谱检索和多智能体流程编排组合起来，让用户可以从“单次 AI 问答”升级到“可沉淀、可编排、可执行、可复用”的数字员工工作流。

平台不是单纯的聊天机器人，而是一个支持 [[多智能体协作]]、[[技能管理]]、[[知识图谱]]、[[GraphRAG]]、[[Agent市场]] 和 [[低代码编排]] 的综合系统。用户可以导入技能、选择角色、配置流程、运行任务、查看中间过程和最终结果，并逐步把优质经验沉淀为可复用能力。

## 2. 解决的问题

传统 AI 使用方式通常停留在一次性问答，存在以下问题：

- 经验难沉淀：提示词、流程、工具调用方法散落在个人对话中，难以团队复用。
- 复杂任务难闭环：一个业务目标往往需要调研、分析、设计、实现、验证、总结等多阶段协作，普通问答无法稳定管理过程。
- 工具与知识割裂：知识库、外部工具、业务流程、Agent 角色各自独立，用户需要手工拼接。
- 垂直场景落地成本高：金融、软件研发、学术研究、产品管理等场景需要专业技能和固定流程，单一通用助手难以胜任。
- 多角色协同不可视：多个 Agent 的分工、依赖、输出与失败状态缺少清晰的可视化编排和追踪。

[[数字员工仿真平台]] 通过 [[Skill沉淀]]、[[多智能体流程]]、[[GraphRAG知识检索]] 和 [[DAG编排]] 解决这些问题。

## 3. 目标用户

- 企业数字化团队：希望构建内部数字员工、自动化知识问答和业务流程助手。
- 产品与项目团队：用于需求调研、PRD 设计、竞品分析、UAT、上线复盘等流程。
- 软件开发团队：用于代码实现、代码评审、测试验证、部署运维、工程方案评审。
- 金融与研究团队：用于财务模型、投研分析、行业研究、客户报告、尽调材料处理。
- 学术与内容团队：用于论文写作、文献综述、同行评审、研究计划和报告生成。
- Agent/Skill 创建者：用于导入、整理、测试和编排可复用技能包。

## 4. 核心功能模块

### 4.1 对话与数字员工助手

对话模块提供面向用户的智能助手入口，支持流式输出、历史会话、模型选择、文件结果下载和知识图谱问答。

能力包括：

- [[SSE流式对话]]
- 会话创建、重命名、删除和历史加载
- 用户记忆与用户画像
- LLM Provider 选择
- GraphRAG 智能搜索
- 工具调用结果展示
- 生成文件下载

### 4.2 Skill 管理

[[Skill管理]] 是平台沉淀能力的关键模块。Skill 可以理解为一组可复用的工作说明、工具脚本、参考资料和执行约束。平台支持把 GitHub 或本地目录中的 Skill 导入系统，并在对话或流程中使用。

能力包括：

- Skill 列表查看
- Skill 启用 / 禁用
- Skill 删除
- ZIP Skill 包上传
- GitHub Skill 导入
- Skill 工具注册
- Skill 元数据展示
- Skill 与多智能体流程联动

典型 Skill 类型包括：

- 软件研发：代码实现、测试驱动开发、系统调试、代码评审、部署发布。
- 产品管理：需求分析、PRD、MVP 设计、竞品分析、客户报告。
- 金融投研：DCF、LBO、三表模型、KYC、组合监控、收益归因。
- 学术研究：论文写作、论文评审、深度研究、学术研究流水线。

### 4.3 多智能体编排

[[多智能体编排]] 是平台区别于普通 AI 问答的核心能力。用户可以创建流程，把多个角色或 Skill 组织成可执行工作流。

支持的流程模式包括：

- [[顺序执行]]：角色按顺序依次处理，上游输出作为下游输入。
- [[并行执行]]：多个角色同时处理同一输入，输出并列展示。
- [[主从协作]]：主控 Agent 拆解任务，多个执行 Agent 完成子任务，再由主控汇总。
- [[竞争评审]]：多个候选 Agent 给出方案，裁决 Agent 汇总比较并选择。
- [[流水线模式]]：按阶段推进，每个阶段形成结构化交付物。
- [[对等协作]]：多个角色互相广播、评审和融合结果。
- [[DAG编排]]：通过画布定义依赖关系，支持复杂非线性工作流。

多智能体页面支持：

- 流程创建、编辑、删除
- 角色选择与排序
- DAG 画布编排
- GraphRAG 节点接入
- 运行过程追踪
- Agent 输出面板
- 协作消息追踪
- 运行产物下载

### 4.4 Skill Tab 与垂直领域角色

[[Skill Tab]] 用于把不同来源的 Skill 按行业或领域组织成角色库。例如：

- software-engineering：软件工程角色
- academic-research：学术研究角色
- finance：金融分析角色
- managed-skills：技能管理页已启用 Skill

用户在创建流程时可以从不同 Skill Tab 中选择角色，形成面向具体场景的工作流。例如学术研究流程可以选择 academic-paper、academic-pipeline、academic-paper-reviewer、deep-research；金融流程可以选择 dcf-model、lbo-model、portfolio-monitoring 等。

### 4.5 知识图谱与 GraphRAG

[[知识图谱]] 模块支持导入 Obsidian vault zip，解析 Markdown、frontmatter、标签、正文标签和 wikilinks，形成知识节点和关系边。

知识图谱能力包括：

- Obsidian vault zip 导入
- 知识源管理
- 节点搜索
- 子图查询
- 邻居关系查看
- 图谱可视化
- GraphRAG 检索
- 基于知识图谱上下文的问答

主要节点类型：

- note：Markdown 笔记
- tag：标签
- entity：未解析 wikilink 或实体
- folder：文件夹结构

主要关系类型：

- links_to：笔记链接到另一笔记
- backlinks_to：反向链接
- has_tag：笔记拥有标签
- mentions：笔记提及实体
- in_folder：笔记位于文件夹

### 4.6 Agent 市场

[[Agent市场]] 用于展示和管理垂类智能体或应用入口。每个 Agent 可以配置标题、功能说明、描述、访问链接、知识库链接、标签、作者、分类和推荐状态。

适用用途：

- 展示企业内部数字员工
- 管理垂直行业智能体
- 沉淀可访问的 Agent 应用
- 连接外部系统入口和知识库入口

### 4.7 Provider 管理

平台支持多个 LLM Provider。管理员可以配置模型供应商、Base URL、API Key、模型列表和自定义 Header。

当前架构支持 OpenAI 兼容协议，并对 DeepSeek 等模型进行适配。对话、多智能体流程和 GraphRAG 问答都可以复用 Provider 能力。

## 5. 典型使用场景

### 5.1 软件研发交付流程

针对产品经理、研发负责人、测试和运维团队，在从需求到上线的场景下，为了解决复杂研发任务缺少结构化协作的问题，平台提供多角色研发流程编排能力。

典型流程：

1. 需求调研：调用调研、竞品、客户审阅类 Skill。
2. 需求分析：使用产品、工程、设计评审角色。
3. PRD / MVP 设计：生成范围、流程、原型和验收标准。
4. 技术开发：调用代码实现、测试驱动开发和系统调试技能。
5. UAT 测试：生成测试用例、验收清单和缺陷分级。
6. 部署运维：生成部署方案、回滚方案和监控指标。

### 5.2 学术研究流程

针对研究人员、学生、论文作者和评审者，在学术写作和研究设计场景下，为了解决研究流程长、材料多、质量标准复杂的问题，平台提供学术 Skill Tab 和多智能体研究流水线。

典型能力：

- 研究问题澄清
- 文献综述
- 论文大纲
- 论文草稿
- 同行评审
- 诚信与引用检查
- 修改建议
- 最终格式化

### 5.3 金融分析流程

针对投研、投行、基金和财务分析用户，在财务建模、投资分析、尽调和报告场景下，为了解决专业模型和材料生成高度依赖经验的问题，平台提供金融 Skill 组合与流程编排。

典型能力：

- 三表模型
- DCF 估值
- LBO 模型
- 可比公司分析
- 收益归因
- 投资建议书
- 投委会备忘录
- KYC 文件解析

### 5.4 企业知识库问答

针对企业内部用户，在制度、项目资料、产品文档、方案沉淀等场景下，为了解决资料分散和检索困难的问题，平台提供 Obsidian 知识库导入、图谱关系构建和 GraphRAG 问答。

用户可以上传 Markdown vault，平台会解析标签、链接和实体关系，形成可视化知识图谱，并支持基于图谱上下文的问答。

### 5.5 数字员工应用市场

针对企业运营和数字化管理人员，在沉淀和推广数字员工应用场景下，平台提供 Agent 市场作为统一入口。用户可以展示各类数字员工、关联访问地址和知识库地址，方便团队发现、使用和维护。

## 6. Agent / AI 能力说明

平台使用的 AI / Agent 能力包括：

- [[大模型理解与生成]]：理解用户需求，生成文档、方案、代码、报告和问答结果。
- [[Agent任务规划]]：把复杂任务拆解为阶段、角色和执行步骤。
- [[工具调用]]：调用内置工具、Skill 工具、知识库检索和文件生成能力。
- [[RAG知识检索]]：基于知识图谱上下文回答问题，减少凭空生成。
- [[多Agent协作]]：多个角色并行、顺序、主从、竞争或对等协作。
- [[自动化执行]]：在流程中自动调用角色、传递上下文、记录输出。
- [[数据分析]]：支持金融、运营、项目、知识图谱等数据分析场景。
- [[代码生成]]：支持软件研发、调试、测试和部署辅助。
- [[内容生成]]：支持 PRD、报告、论文、演示材料、总结文档。
- [[人机协同]]：用户在关键节点选择角色、调整流程、确认输出。

Agent 在平台中承担的角色：

| Agent 角色 | 是否使用 | 说明 |
|---|---|---|
| 助手 | 是 | 完成信息整理、问答、内容生成和用户支持 |
| 分析师 | 是 | 理解数据、提炼洞察、生成分析建议 |
| 执行者 | 是 | 调用工具、执行 Skill、完成流程任务 |
| 协作者 | 是 | 与用户和其他 Agent 共同完成复杂任务 |
| 流程编排者 | 是 | 拆解任务、规划步骤、调度工具或多个 Agent |

与普通 AI 问答相比，平台的新设计在于：

- 能完成连续任务，而不是只回答单轮问题。
- 能主动拆解任务，形成流程和角色分工。
- 能调用工具、Skill、知识库和多智能体服务。
- 能连接业务流程，从输入到输出形成闭环。
- 能保存流程、沉淀 Skill、复用角色和知识。
- 有人机协同机制，用户可以配置、选择、确认和复盘。

## 7. 技术架构

平台采用前后端分离、网关聚合、Agent 服务独立、编排服务独立的架构。

```text
Web React/Vite
  |
  v
Go Gateway / Gin
  |-- Auth / JWT / User / Conversation
  |-- Chat API / SSE
  |-- Skill API Proxy
  |-- Marketplace API
  |-- Knowledge Graph Proxy
  |-- Hermes Bridge Proxy
  |
  +--> Python Agent / FastAPI / LangGraph
  |      |-- Chat Graph
  |      |-- Skill Manager
  |      |-- Tool Registry
  |      |-- LLM Provider Adapter
  |      |-- Memory
  |      |-- Knowledge Graph / GraphRAG
  |
  +--> Hermes Bridge / FastAPI
         |-- Expert Roles
         |-- Skill Tabs
         |-- Dialog Flows
         |-- Flow Runs
         |-- Multi-Agent Orchestrator
```

主要技术栈：

| 层级 | 技术 |
|---|---|
| 前端 | React、Vite、TypeScript、Ant Design、Zustand |
| 网关 | Go、Gin、JWT、SSE、SQLite |
| Agent 服务 | Python、FastAPI、LangGraph |
| 多智能体编排 | Hermes Bridge、Flow Orchestrator |
| 知识图谱 | Markdown/Obsidian 解析、节点边存储、GraphRAG |
| 数据存储 | SQLite |
| 部署 | Caddy、Linux 服务进程、静态资源目录 |

## 8. 核心数据流

### 8.1 普通对话

1. 用户在 Web 输入问题。
2. Go Gateway 校验 JWT。
3. Gateway 创建或加载会话。
4. Gateway 通过 SSE 转发到 Python Agent。
5. Python Agent 调用 LLM、工具、Skill 或知识检索。
6. Agent 流式返回中间事件和最终结果。
7. Gateway 保存消息并推送给前端展示。

### 8.2 多智能体流程

1. 用户创建流程，选择流程模式和角色。
2. Web 调用 `/api/v2/flows` 保存流程。
3. 用户输入任务并启动 run。
4. Hermes Bridge 根据流程类型调度多个角色。
5. 每个角色执行自身 Skill 或提示模板。
6. 编排器收集中间结果、协作消息和失败状态。
7. 前端展示每个角色输出、运行状态和产物下载。

### 8.3 知识图谱导入

1. 用户上传 Obsidian vault zip。
2. Python Agent 解析 Markdown 文件。
3. 提取 frontmatter、tags、wikilinks、aliases 和正文。
4. 写入 note、tag、entity、folder 节点。
5. 写入 links_to、has_tag、mentions、in_folder 等关系。
6. 用户可以搜索节点、查看子图、使用 GraphRAG 问答。

## 9. 权限与安全

平台使用 JWT 登录态进行 API 鉴权。主要安全设计包括：

- 注册、登录、个人信息接口。
- 生产环境禁用开发登录。
- API 网关统一校验 Authorization。
- 前端遇到 401 自动清理过期 token 并要求重新登录。
- 管理员可维护 LLM Provider。
- CORS、安全响应头、请求恢复中间件和基础限流。

当前需要持续优化的方向：

- Skill 管理类写操作建议进一步强化管理员权限。
- Hermes Bridge 直连服务建议保持内网访问。
- 多租户知识图谱隔离可作为后续企业化能力。
- Provider API Key 应持续避免进入日志、仓库和前端响应。

## 10. 预期价值

平台落地后可带来的价值包括：

- 提升效率：把多步骤工作从人工串联变成可复用流程。
- 降低成本：减少重复提示词编写、材料整理和人工协调。
- 提升质量：通过多角色评审、知识检索和流程化检查减少遗漏。
- 沉淀能力：把优秀经验沉淀为 Skill、流程和知识图谱。
- 支持规模化：通过 Agent 市场和 Skill Tab 推广到更多团队和场景。
- 改善体验：用户从“找 AI 聊天”变成“调用数字员工完成任务”。
