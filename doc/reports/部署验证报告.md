## 五、部署过程中发现/确认的问题

### 🟡 部署过程中的坑
1. **首次 `cp -av` 报 `Text file busy`**：无法覆盖正在运行的 ELF 二进制。→ 必须先 kill 原进程再 swap binary。
2. **`pkill -f 'python main.py'` 未命中**：因为 Python 进程 cmdline 被 uvicorn reloader 改写成 `/venv/bin/python -c 'from multiprocessing.spawn...'`，不含 `main.py` 字符串。→ 改用 `fuser -k 8001/tcp`。
3. **`nohup ... & disown` 挂住 SSH 通道**：paramiko 等待 exit status 无果。→ 改用 `( CMD & )` 子 shell fork 模式。
4. **gateway 日志尾部有 `bind: address already in use`**：第 3 次尝试 start 时的冗余错误，实际是第 2 次 start 已抢占端口；现网运行的 gateway (pid 100954) 是正确的新实例。

### 🟡 服务器侧的既存问题（未动）
1. **Python 3.6.8 系统版 + 3.11.15 venv 并存**：hermes-bridge 的 `source venv/bin/activate` 静默失败，回退到系统 3.6 运行；建议后续排查。
2. **`/root/projects/agent-platform/` 是一份孤立的、不完整的旧快照**（6 目录，无 build/.env/hermes-bridge）：与本次部署无关，但占空间，建议用户自行决定是否删除。
3. **gateway `/health` 无显式路由**：`GET /health` 落入 SPA 静态托管，返回 index.html 而非 JSON。存活探测应走 `/api/v1/agent/health` 或通过进程/端口监控。
4. **正式服未配置 hermes-gateway 以外的 systemd 单元管理 agent/gateway**：两个服务都是 `bash -lic` 后台 fork，掉电/重启后不会自动拉起。
