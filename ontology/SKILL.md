# 本体驱动贷款预审助手

基于知识本体与大模型，自动从制度文件和历史审批报告中抽取结构化知识，并对新贷款申请生成可解释、可追溯的预审意见。

## 工作模式

本技能支持三种工作模式，根据用户意图自动切换：

| 模式 | 触发场景 | 说明 |
|------|----------|------|
| **`ingest`** | 用户提供制度文件/审批报告，要求建立或更新知识体系 | 文档解析 → 本体抽取 → 知识入库 |
| **`update`** | 用户提供新制度条款，要求纳入已有本体 | 增量抽取 → 冲突检测 → 本体更新 |
| **`review`** | 用户提供贷款申请材料，要求生成预审意见 | 申请解析 → 本体匹配 → 规则推理 → 预审报告 |
| **`visualize`** | 用户要求查看/展示本体结构、规则分布或知识图谱 | 读取本体 JSON → 生成交互式 HTML 可视化图 → 上传 BOS |

## 模式一：知识入库（ingest）

### 步骤

1. **对象链层级定位**（新增，参见 `references/object-chain-guide.md`）
   - 判断输入材料属于四层对象建设链的哪一层：
     - 以**制度条款/审批规则**为主 → 第2层（本体语义层，主战场，重点抽取 rules + concepts）
     - 包含**组织职责/流程边界** → 补充抽取第1层归属关系，写入 `concepts.GovernanceStructure`
     - 包含**历史案例/关联事件** → 同时构建第3层实例关系，扩展 `relations` 字段
   - 向用户说明本次材料的层级定位，以及后续建议补充哪层材料

2. **识别输入材料类型**
   - 风险管理制度文件（规则来源）
   - 历史审批报告（规则例证/补充）
   - 支持格式：用户粘贴的文本、附件路径（txt/pdf/md/docx）

3. **文档分段与预处理**
   - 按章节/条款粒度拆分，单段不超过 800 字
   - 标记段落类型：定义类、规则类、示例类、例外类

3. **本体要素抽取**（调用大模型，使用 `references/ontology-schema.md` 中的 Schema）
   - **核心概念**：客户（借款人）、担保（抵押/质押/保证）、财务指标、贷款产品
   - **审批规则**：准入条件、限制条件、否决条件、例外条款
   - **关系**：适用关系、依赖关系、互斥关系、继承关系
   - **属性**：阈值、计量单位、时间窗口、适用范围

4. **本体存储**
   - 将抽取结果以结构化 JSON 写入 `/home/gem/workspace/.ark/tmp/ontology_kb.json`
   - 同步输出人类可读摘要至 `/home/gem/workspace/.ark/output/<任务子目录>/本体知识摘要.md`

5. **抽取结果确认**
   - 向用户展示抽取的核心概念数量、规则条目数量
   - 列出置信度较低（<0.8）的条目，请用户确认或修正

## 模式二：增量更新（update）

1. **解析新条款内容**，识别变更类型：新增、修改、废止

2. **冲突检测**：与已有本体条目比对，识别：
   - 直接冲突（新旧规则结论相悖）
   - 覆盖关系（新条款替代旧条款）
   - 补充关系（新条款细化已有规则）

3. **更新决策**：对每个冲突点给出建议处理方式，经用户确认后执行

4. **版本记录**：在本体文件中追加变更日志，记录条款来源、生效日期、变更摘要

## 模式三：贷款预审（review）

### 步骤

1. **申请材料解析**
   - 提取申请人基本信息、财务数据、担保安排、贷款用途

2. **本体匹配**
   - 将申请要素映射到本体中的概念节点
   - 加载适用的审批规则集（见 `references/review-rules.md`）

3. **规则推理**（逐条执行，保留推理路径）
   - 准入条件核验（硬性门槛）
   - 财务指标评估（定量指标与阈值比对）
   - 担保充足性评估
   - 否决条件扫描
   - 例外条款识别

4. **预审意见生成**（严格按 `references/review-output-template.md` 格式输出）
   - 初审结论：通过 / 有条件通过 / 不通过
   - 主要风险点（每项附对应制度条文编号）
   - 需核实/补充事项
   - 推理路径摘要（每个结论可溯源至制度条款）

5. **输出交付**
   - 将预审报告写入 `/home/gem/workspace/.ark/output/<任务子目录>/贷款预审报告.md`
   - 在对话中展示摘要版本（核心结论 + 前 3 项风险点）

## 模式四：本体可视化（visualize）

### 触发场景

用户说"展示本体"、"可视化知识图谱"、"显示规则分布"、"画出本体结构"等。

### 步骤

1. **加载本体数据**
   - 读取 `/home/gem/workspace/.ark/tmp/ontology_kb.json`
   - 若文件不存在，提示用户先执行 `ingest` 模式导入制度文件

2. **生成可视化**，调用脚本：
   ```bash
   python3 scripts/visualize_ontology.py \
       --kb /home/gem/workspace/.ark/tmp/ontology_kb.json \
       --output /home/gem/workspace/.ark/output/<任务子目录>/ontology_viz.html
   ```
   脚本生成包含三个视图的交互式 HTML 页面：
   - **概念图谱**（节点图）：4 大概念节点 + 规则节点，颜色区分类型，边表示关系
   - **规则分布图**（饼图+柱状图）：按 veto / condition / threshold / exception 分类统计
   - **条款来源分布**（条形图）：按来源文件统计规则数量

3. **交付**
   - 上传 HTML 到 BOS：`dodo_cli bos upload --short-url --no-ath <文件路径>`
   - 在对话中给出可直接打开的链接

## 重要原则

- **可解释性优先**：每条结论必须标注对应的制度条文来源，不允许出现"根据综合判断"等无法追溯的表述
- **置信度标注**：对信息不足或模糊的判断项，标注"待核实"，不强行给出确定性结论
- **知识版本意识**：本体文件不存在时，提示用户先执行 `ingest` 模式导入制度文件
- **隐私保护**：处理申请人个人信息时，不将数据写入公开可访问位置，不在对话摘要中展示完整证件号/账号等敏感字段
