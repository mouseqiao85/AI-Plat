# 四层对象建设链落地指引

> 来源：企业架构研究会《为什么做了业务架构，还需要继续走向本体论、知识图谱并完善业务对象数字化》
> 作者：侯君璠（AI架构陪跑师）· 2026年4月20日
> 知识引入版本：loan-ontology-reviewer v1.1

---

## 核心思想

**四层对象建设链**是企业对象体系从治理到运行的完整演进路径，四者不是并列工具，而是逐层递进的依赖关系：

```
业务架构  →  本体论  →  知识图谱  →  业务对象数字化
  (治理边界)   (正式语义)  (实例网络)    (运行系统)
```

> 业务架构定义治理边界，本体论定义正式语义，知识图谱承载实例关系网络，
> 业务对象数字化让这些对象真正进入主数据、接口、知识库与智能应用。

---

## 各层定义与核心问题

| 层级 | 偏向 | 核心问题 | 若只做这层的缺陷 |
|------|------|----------|-----------------|
| **业务架构** | 治理与结构 | 对象归属哪个能力域？协同边界是什么？ | 对象只获得业务归属，缺乏正式语义 |
| **本体论** | 语义与约束 | 对象的概念/属性/关系/约束是什么？ | 语义完整但未进入实例网络，无法支撑关系推理 |
| **知识图谱** | 实例与网络 | 对象之间如何连接？多跳关系如何穿透？ | 关系丰富但缺少治理边界和语义约束，容易漂移 |
| **业务对象数字化** | 工程与运行 | 对象如何进入数据/接口/知识库/智能系统？ | 退化为字段映射和接口拼装，失去上层治理和语义基础 |

## 适用场景判断

使用本技能处理文档时，应先判断输入材料的层级属性：

### 何时重点做本体抽取（第2层）
- 输入材料以**制度条款、审批规则、风险管理规定**为主
- 存在**概念不一致、术语漂移、规则解释冲突**的问题
- 目标是建立跨系统、跨部门的**共享语义资产**
- ✅ 这是 `loan-ontology-reviewer` 的主要工作场景

### 何时同时构建知识图谱（第3层）
- 输入材料包含**历史审批案例、关联事件记录**
- 需要**多跳关系检索**（如：某客户 → 关联企业 → 历史违约记录 → 适用规则）
- 已有 GraphRAG 或 Agent 的下游需求

### 何时补充业务架构层（第1层）
- 输入材料涉及**组织架构调整、职责边界变化**
- 需要明确**审批规则归属哪个能力域/部门**
- 检测到规则间存在**治理职责冲突**（而非语义冲突）

## 工程落地路径（以主题域为单位）

不建议一开始追求全企业统一模型，推荐**主题域迭代推进**策略：

### Step 1：选定高价值对象域
选择一个核心主题域开始，例如：
- 贷款预审：**客户** → **担保** → **财务指标** → **审批规则**
- 保险核保：**被保险人** → **标的** → **核保规则** → **费率**

成果：对象目录、对象归属表、边界清单（不是复杂图，而是清单）

### Step 2：建立正式语义（本体层）
**从业务含义出发**，而非从数据库字段或旧系统表结构反推概念：
- ✅ 正确：先定义"客户"在风险管理语境下的含义，再映射到数据模型
- ❌ 错误：直接用 `customer_table` 的字段列表代替概念定义

关键产出：`ontology_kb.json` 中的 `concepts` + `rules` 字段

### Step 3：引入知识图谱（按需，非必须）
**仅在确实存在多跳关系需求时**才引入，图谱价值在于承载有用的实例关系网络，而非规模：
- 判断标准：是否需要"客户A → 关联企业B → 违约记录C → 适用规则R"这类推理链
- 产出：`ontology_kb.json` 中的 `relations` 字段扩展

### Step 4：业务对象数字化（接入运行系统）
确保同一个对象在各运行层保持**同一身份、同一含义、同一关系语境**：
- 主数据系统：对象 ID 统一
- 接口契约：字段语义一致
- 知识库/RAG：标签和检索索引与本体概念对齐
- Agent 工具：规则推理路径可追溯至本体条款

## 贷款预审场景的四层映射

| 层级 | 贷款预审具体内容 |
|------|----------------|
| 业务架构层 | 信贷审批能力域划分、审批职责归属（风控部/合规部/业务部）、价值流节点 |
| **本体语义层** ← 主战场 | 客户、担保、财务指标、审批规则的概念定义；准入/否决/例外条款的语义约束 |
| 知识图谱层 | 历史审批案例图谱；客户-担保-违约记录关联网络；规则间依赖/冲突关系 |
| 业务对象数字化层 | 本体JSON接入主数据；规则接口化供审批系统调用；知识库向量化供RAG检索 |

## 常见反模式（避免）

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 从数据库字段反推本体概念 | 旧系统命名混乱会污染本体语义 | 先从业务含义定义，再映射到数据 |
| 一开始就建全企业统一知识图谱 | 边界不清，规模失控，维护成本高 | 以主题域为单位，逐步收紧 |
| 只做本体定义不做业务对象数字化 | 停留在模型层，无法进入运行系统 | 必须推进到第4层才能产生实际价值 |
| 跳过本体直接建知识图谱 | 缺少语义约束，关系网络容易漂移失真 | 先完成第2层本体语义，再扩展第3层 |
