## 推理执行顺序

预审推理按以下固定顺序执行，**不可跳过任何阶段**：

```
阶段1: 一票否决扫描（veto rules）
  ↓ 任一触发 → 直接输出"不通过"，记录触发规则，停止后续阶段
阶段2: 准入条件核验（threshold rules / condition rules）
  ↓ 全部通过 → 继续
  ↓ 部分不通过 → 标记风险项，继续后续阶段（不立即否决）
阶段3: 财务指标评估（financial threshold rules）
  ↓ 逐项比对，记录满足/不满足/待核实
阶段4: 担保充足性评估
  ↓ 按担保类型加载对应规则，评估是否满足
阶段5: 例外条款扫描（exception rules）
  ↓ 检查前序阶段中不满足的项是否有例外条款可豁免
阶段6: 综合研判
  ↓ 汇总各阶段结论，输出最终初审意见
```

---

## 规则优先级

| 优先级 | 规则类型 | 说明 |
|--------|----------|------|
| P0 | `veto` | 一票否决，触发即拒绝，不受例外条款影响 |
| P1 | `condition` | 复合条件，满足所有子条件才算通过 |
| P2 | `threshold` | 数值阈值，超出范围标记风险 |
| P3 | `exception` | 例外豁免，可覆盖 P2 级规则的不满足结论 |

**注意**：例外条款（P3）**不能**覆盖一票否决（P0）规则。

## 置信度机制

每次推理结论附带置信度说明：

| 置信度 | 含义 | 处理方式 |
|--------|------|----------|
| ≥ 0.9 | 高置信，信息充分 | 直接给出结论 |
| 0.7 ~ 0.9 | 中等置信，信息基本完整 | 给出结论，附"建议核实"标注 |
| 0.5 ~ 0.7 | 低置信，信息存在缺口 | 标注"待核实"，不做确定性判断 |
| < 0.5 | 信息严重不足 | 标注"无法判断"，要求补充材料 |

**置信度降低的常见原因**：
- 申请材料未提供某字段的数据
- 本体库中对应规则的 `confidence < 0.8`
- 申请数据与规则的适用范围不完全匹配
- 存在多条相互矛盾的规则

## 冲突与例外处理

### 规则冲突处理

当本体库中存在两条对同一指标得出相悖结论的规则时：

1. 比较规则的 `effective_date`，**优先采用更新的规则**
2. 若日期相同，比较规则的 `confidence`，采用置信度更高的
3. 若均无法区分，**两条规则同时标记**，降低整体置信度，要求人工复核

### 例外条款匹配

例外条款的适用需同时满足：
- 例外条款的 `applies_to` 覆盖当前申请人的类型
- 申请人满足例外条款的触发条件（如"属于国家支持的XX行业"）
- 所豁免的规则在例外条款的 `exception_of` 字段中有明确引用

**例外条款匹配后**：在预审报告的"五、例外条款适用检查"中明确列出，不隐式影响结论。

## 推理路径记录规范

每条推理路径记录以下信息，格式化为可读的追溯链：

```
[申请字段值]
→ 匹配规则 [rule_id]（[来源文件] [条款编号]）: [规则描述]
→ 条件: [字段] [运算符] [阈值] [单位]
→ 实际值: [申请中的值]
→ 比较结果: [满足/不满足/无法判断]
→ 规则动作: [pass/reject/flag/require_supplement]
→ 置信度: [0.xx]
```

**示例**：

```
资产负债率: 68%
→ 匹配规则 R-001（风险管理办法2024版 第三章第十二条）: 企业客户资产负债率不得超过70%
→ 条件: asset_liability_ratio <= 70 %
→ 实际值: 68%
→ 比较结果: 满足 ✅
→ 规则动作: pass
→ 置信度: 0.98
```

```
利息保障倍数: 1.8
→ 匹配规则 R-007（风险管理办法2024版 第三章第十五条）: 利息保障倍数不低于2.0
→ 条件: interest_coverage >= 2.0
→ 实际值: 1.8
→ 比较结果: 不满足 ❌
→ 规则动作: flag → 生成风险点"利息保障倍数不足"
→ 置信度: 0.95
```
