## 一、一句话定义

让平台用户**勾选多个 gstack 专家角色 + 选一个工具场景 + 触发一次多智能体协作**，由后端按编排策略调度各角色并行/串行工作，最终**按角色分栏汇总结果**呈现给用户并可导出。

---

## 二、核心用户流程（MVP）

```
1. 用户进入"多智能体协作"入口
2. 用户在角色面板勾选 1~N 个 gstack 专家角色（多选）
3. 用户从下拉选一个"工具场景"（preset：code-review / product-discovery / design-audit / release ...）
4. 用户填写任务描述（message），点击"开始协作"
5. 前端打开 SSE 连接，后端按编排策略跑多角色协作
6. 前端按角色分栏实时渲染：每个角色 = 一个卡片，显示自己的 markdown 输出 + 引用 + 产物
7. 协作完成后，用户可一键导出整份报告（Markdown / PDF）
```

### 入口与导航
- 在左侧主导航增加 "多智能体协作" 入口（与 Chat / Skills / Agents 同级）
- 对话历史按 `collaboration_session_id` 分组，区别于普通单轮对话

## 三、架构决策（锁定）

| 决策项 | 选择 | 备注 |
|---|---|---|
| **Subagent 路径** | **C. 混合** | install 走 hermes `skills/hub/install`；单角色调用走 agent 服务 LangGraph Worker（通过 `skill_<name>_run` tool 调 hermes CLI） |
| **23 角色安装时机** | **启动时全量预装** | 新增 `scripts/deploy/preinstall_gstack_roles.sh`，部署时跑一次批量 copytree + reload；后续热重载 |
| **工具场景定义位置** | **`configs/tool_scenarios.yaml`（后端静态）** | 版本随代码走；改动需发版；初期预设 5~8 个场景 |

### 多角色协作编排（混合路径的具体形态）
- **默认策略：并行 + 汇总**
  - 所有勾选的角色同时跑，各自独立 Worker session
  - 最后由一个 "coordinator" 节点汇总
  - 适合"多个视角平行评审"类任务（code-review 多人看、design 多人打分）
- **可选策略：串行 handoff**（v1+，非 MVP）
  - 角色按用户指定顺序执行，前者输出作为后者输入
  - 适合"流水线"类（CEO plan → eng review → design review → ship）
- **可选策略：supervisor**（v2+）
  - 引入一个主管节点，动态决定调度顺序

**MVP 只交付"并行 + 汇总"**。串行/supervisor 作为后续迭代。

## 四、后端改造点

### 4.1 Python Agent 服务（`agent/`）

#### 4.1.1 新增 `RoleWorker`
位置：`agent/app/workers/role_worker.py`
- 继承或并列于现有 `Worker`
- 入参：`role_slug`（gstack skill name）、`task`（用户原始 message）、`scenario`（工具场景名）、`shared_context`（其他角色的中间产出，v1+）
- 实现：构造 prompt → 调 `skill_<role_slug>_run` tool → 解析结果为 `RoleReport`
- 输出 `RoleReport(role_slug, markdown, citations, artifacts, duration, status)`

#### 4.1.2 新增 `CollaborationOrchestrator` 节点
位置：`agent/app/graph/nodes/collaboration.py`
- 输入：`AgentState.collaboration_request = { roles: [...], scenario: "..." }`
- 行为：用 `asyncio.gather` 并发启动 N 个 RoleWorker（N ≤ 10，上限提到和角色数匹配）
- 输出：`state.role_reports = [RoleReport, ...]`

#### 4.1.3 新增图分支
修改 `agent/app/graph/graph.py`：
- `router` 节点识别 `collaboration_request` 是否非空
- 若非空 → 跳过 planner/worker_orchestrator → 直接进入 `collaboration` 节点 → responder
- 若为空 → 原路径不变

#### 4.1.4 工具场景注册
新增 `configs/tool_scenarios.yaml`（注意：放在 agent 服务目录 `agent/configs/` 或 repo 顶层，二选一）：
```yaml
scenarios:
  - name: code-review
    display_name: 代码评审
    description: 适合代码/架构评审类协作
    tools:
      - skill_review_run
      - skill_investigate_run
      - brave_search
  - name: product-discovery
    display_name: 产品发现
    description: 适合产品定义 / CEO 视角探索
    tools:
      - skill_office-hours_run
      - skill_plan-ceo-review_run
      - brave_search
  - name: design-audit
    display_name: 设计审查
    description: UI/UX 设计打分与改进
    tools:
      - skill_design-review_run
      - skill_design-shotgun_run
      - skill_plan-design-review_run
  # ... 5-8 个
```

`agent/app/tools/registry.py` 增加：
- `load_scenarios(path)` 启动时加载
- `get_tools_for_scenario(name) -> list[Tool]`
- `list_scenarios()`

#### 4.1.5 SSE 事件扩展
当前事件（`ping/text/card/file_download/plan/worker_*`）新增：
- `role_started` `{role_slug, order}`
- `role_progress` `{role_slug, partial_markdown}`（可选，v1+ 真流时）
- `role_done` `{role_slug, role_report}`
- `collaboration_done` `{summary, artifacts}`

### 4.2 Go Gateway（`internal/`）

#### 4.2.1 新增路由
- `GET /api/v1/collaboration/roles` — 列出预装的 gstack 23 角色（元数据从静态 yaml 或从 agent 服务拉）
- `GET /api/v1/collaboration/scenarios` — 列出工具场景
- `POST /api/v1/collaboration/sessions` — 创建协作会话（落库 `collaboration_sessions` 表）
- `POST /api/v1/collaboration/sessions/:id/stream` — SSE 流式跑协作
- `GET /api/v1/collaboration/sessions/:id` — 拿会话详情+所有 RoleReport
- `POST /api/v1/collaboration/sessions/:id/export` — 导出 Markdown/PDF

#### 4.2.2 新增 handler
`internal/api/handler/collaboration.go`
- 透传协作请求到 agent 服务的 `POST /api/v1/agent/collaboration/stream`
- 或直接内部包装 agent 服务现有 `POST /api/v1/chat/stream`，body 里加 `collaboration_request`

#### 4.2.3 DB schema
新建表：
```sql
CREATE TABLE collaboration_sessions (
  id INTEGER PRIMARY KEY,
  user_id INTEGER NOT NULL,
  title TEXT,
  task TEXT NOT NULL,
  roles TEXT NOT NULL,       -- JSON array of slugs
  scenario TEXT NOT NULL,
  status TEXT NOT NULL,      -- pending|running|done|failed
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

CREATE TABLE role_reports (
  id INTEGER PRIMARY KEY,
  session_id INTEGER NOT NULL,
  role_slug TEXT NOT NULL,
  markdown TEXT,
  citations TEXT,             -- JSON
  artifacts TEXT,             -- JSON
  status TEXT,
  duration_ms INTEGER,
  created_at TIMESTAMP
);
```

### 4.3 Hermes-Bridge & 预装脚本

#### 4.3.1 新增 `scripts/deploy/preinstall_gstack_roles.sh`
- 从 `C:\Projects\gstack\{role}/SKILL.md`（或服务器克隆的 gstack 仓库）**批量 install** 23 个 skill 到 `/home/admin/agent-platform/agent/skills/`
- 完成后 POST `http://127.0.0.1:8001/api/v1/skills/admin/reload`
- 部署流水线：首次 deploy 跑一次；之后 gstack 版本升级时重跑
- **幂等**：已存在的 skill 先 rm 再 copy

#### 4.3.2 Bridge 契约不动
MVP 不走 bridge 的多角色 endpoint，所有多角色调度在 agent 服务侧做。bridge 只用 `/api/v2/skills/{name}/execute`（单 skill）。

## 七、非功能需求

### 性能
- 并发角色 ≤ 10（提上限到匹配需求，现有 MAX_CONCURRENT_WORKERS=5 不够）
- 单角色 hermes CLI 执行 ≤ 300s（沿用 bridge timeout）
- 整个协作会话超时 = max(单角色 timeout) + 30s

### 鉴权
- 所有 `/api/v1/collaboration/*` 路由走 `AuthMiddleware`
- `collaboration_sessions.user_id` 强制隔离，用户只能查自己的会话
- 导出的 PDF/Markdown 不带其他用户数据

### 可观测
- 每次 RoleWorker 执行记录到 `role_reports` 表 + 日志
- Gateway 侧打 metric：`collaboration.roles.count`、`collaboration.duration.seconds`、`collaboration.role.failure.count{role_slug}`

### 错误处理
- 单个角色失败 → `role_reports.status=failed`，其他角色继续，最终 summary 显示失败项
- 全部失败 → session status=failed，给用户清晰错误
- Hermes CLI not found / skill 未装 → 提前检查，报明确错误（不是 500）

## 八、风险与开放问题

### 风险
1. **Bridge 伪 SSE 会放大 UX 差**：并行 10 角色每个都卡到完成才切片回传，用户可能等 1~5 分钟无真实反馈。M2 必须真流。
2. **Hermes CLI 输出解析脆弱**：skill 输出不是结构化 JSON，citations/artifacts 提取要正则 + 约定，易失效。
3. **Skill 列表解析脆弱**：`hermes_cli.py` 的表格解析。MVP 绕过 → 用静态角色元数据表，不依赖 `hermes skills list` 运行时解析。
4. **并发压测未做**：LLM API / hermes CLI 同时跑 10 个的实际 CPU/内存占用未验证。M1 收尾前需做一次压测。
5. **23 角色在服务器预装**：部署脚本失败的处理路径、gstack 仓库版本锁定方式都未定（选 tag？分支？commit sha？）

### 开放问题（需要产品/设计侧回答）
- Q1: 用户一次协作的平均角色数预期？（影响并发上限决策）
- Q2: 结果导出是否需要带 watermark / 企业 branding？
- Q3: 角色元数据（头像、颜色）谁来出？设计师还是用 emoji 占位？
- Q4: "工具场景"的英文名用 scenario 还是 preset / toolset / workflow？影响 UI 翻译一致性
- Q5: 协作会话的分享链接需求（只读 URL 给团队看）是否在 MVP 范围内？

### 与现有系统冲突点
- 现有 `selectedSkill` 单选逻辑保留（单技能对话走 chat），新 `selectedRoles` 多选只在协作页生效 —— **两条路径并存，别互相污染**。
- `worker_orchestrator` 的 5 并发限制会影响单轮对话的 planner 产出的 step 数量；调高会同时影响普通对话性能，需谨慎（建议新节点独立参数，别共用 MAX_CONCURRENT）。
