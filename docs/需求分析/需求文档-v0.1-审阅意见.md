## Blocker（不修正 MVP 无法开工）

### 1. **`skill_<slug>_run` 契约与现有代码互相矛盾，等于没有 hermes 调用通道**
- **引用**：v0.1 §三「Subagent 路径 / 混合」、§4.1.1「RoleWorker 调 `skill_<role_slug>_run` tool」、§4.1.4 `tool_scenarios.yaml` 里把 `skill_review_run / skill_office-hours_run` 当 hermes 角色调用点。
- **问题本质**：
  - `agent/app/tools/skill_tools.py:51-125` 的 `RunSkillScriptTool` 是把 `SKILL.md` 目录下的**本地 Python 脚本**读进来，在受限 `exec()` 命名空间里跑（还显式 ban 了 `subprocess / os / open`）。它和 hermes CLI 完全无关。
  - gstack 23 个 skill 大多数**没有可执行 Python script**，它们是 SKILL.md prompt + 参考文档。预装 23 个角色后，`RunSkillScriptTool.execute()` 会命中 `script_path` 不存在或 `script blocked: blocked builtin call: open` 立即失败。
  - 真正会跑 hermes CLI 的只有 `hermes-bridge/bridge/hermes_cli.py:64 execute_skill()`，而 v0.1 §4.3.2 明确写「MVP 不走 bridge 的多角色 endpoint，所有多角色调度在 agent 服务侧做。bridge 只用 `/api/v2/skills/{name}/execute`（单 skill）」——那么 agent 服务侧调什么？
- **最小修正**（二选一，必须在 M1 kickoff 前拍板并写进文档 §三）：
  - **方案 A（推荐）**：新增 `agent/app/tools/hermes_role_tool.py:HermesRoleTool`，name 仍为 `skill_<slug>_run`，内部走 `httpx.post(bridge + "/api/v2/skills/{slug}/execute", json={"message": task})`。bridge 作为唯一 hermes 出口，agent 只做编排。预装脚本不再需要把 skill 目录 copytree 到 `agent/skills/`（那是给 RunSkillScriptTool 用的，hermes 调用不需要）。
  - **方案 B**：RoleWorker 不经 tool 抽象，直接在 `role_worker.py` 里 `httpx` 调 bridge。放弃「所有角色都是 tool」的对称性，换取路径最短。
  - **不可行方案**：沿用 RunSkillScriptTool — 会在 M1 第一天失败。

### 2. **MVP 并发 10 在当前 bridge + hermes CLI 形态下会击穿服务器**
- **引用**：v0.1 §七「并发角色 ≤ 10」、§4.1.2「asyncio.gather 并发 N 个 RoleWorker」、§八「并发压测未做」。
- **问题本质**：
  - `hermes_cli.py:18-25` 每次调用都是 `subprocess.run(["hermes", ...])` 冷启动一个 Node 进程（hermes 基于 `/home/admin/.hermes/node/bin`），空载内存 ~150-250MB，10 个并发 ≈ 2-3GB 常驻；若 skill 还加载 Claude SDK + 浏览器就更炸。
  - bridge 的 `chat_handler.py:23 run_in_executor(None, ...)` 默认走 FastAPI 的 `ThreadPoolExecutor`，**max_workers=min(32, cpu+4)**。10 个并发 subprocess 会占满线程池同时被 `subprocess.run` 阻塞，期间任何其他 bridge 请求（list_skills、hub install）排队 → 部署/健康检查会超时假死。
  - 更关键：gstack 角色背后跑的是 Claude API，单账号常见的 tier-1 并发上限是 5 rpm / 40 requests-per-min（不同 plan 不同），10 并发 × 每角色可能 3-10 次 LLM 调用 = 瞬间 30-100 req，触发 429 后 hermes CLI 默认重试会把单角色拖到 60s+；v0.1 §七「整个协作会话超时 = max(单角色 timeout) + 30s = 330s」会连锁失败。
- **最小修正**：
  1. 把 MVP 目标从「并发 10」降到「并发 3-4」，在 `RoleWorker` 层用独立 `asyncio.Semaphore(COLLAB_MAX_CONCURRENT=3)`（独立于 worker_orchestrator 的 5）。
  2. 在 M1 里安排 1 天做基线压测（1/3/5 并发各跑 3 次），记录峰值 RSS、Claude 429 率、端到端 p50/p95。压测结果 → 上限才敢调高。
  3. bridge 侧加一个 global `asyncio.Semaphore(BRIDGE_MAX_HERMES=4)` 防御性限流，防止 agent 写错并发值直接打穿。
  4. v0.1 §七「并发 ≤ 10」改写为「MVP: 3；M2 压测后决定上限」。

### 3. **伪 SSE 在 10 角色并行场景下用户体验是「空白 3-5 分钟 + 一次性闪屏」**
- **引用**：v0.1 §八「风险 1: Bridge 伪 SSE 会放大 UX 差」、§6 M2「真流式」。
- **问题本质**：读 `chat_handler.py:33-45`，`chat/stream` 是**先 `run_in_executor` 跑完整个 CLI（单角色最长 300s）、拿到完整 response 后才开始 50 char 一片、10ms 一次往外切**。这不是流，是「跑完→伪装成流回放」。
  - 10 角色并行时，agent 侧如果每个 RoleWorker 单独调 bridge `/skills/{name}/execute`（非 stream），就没有任何中间事件；前端看到的是 ≥30s（单角色最短）到 300s（最长）之间**纯空白**，然后某一刻突然冒出 N 个角色卡片齐刷刷滚完 markdown。
  - 加之 `responder` 节点要等所有 worker 完成才能汇总，首字节时间（TTFT）= slowest 角色的 end-to-end 时间 ≈ p95 ≥ 120s。用户在 120s 空白页面的流失率业内普遍 >60%。
- **最小修正**（MVP 必须做一项，不能等 M2）：
  1. agent 侧 `CollaborationOrchestrator` **在每个 RoleWorker 开跑前和完成时立刻 emit `role_started` / `role_done` SSE**（v0.1 §4.1.5 有写，但要求它必须在 M1 就上，不能延到 M2）。这样即使单角色内没有流，用户起码每隔几秒就看到 `1/10 完成... 2/10 完成...` 的进度条刷新。
  2. 对单角色 markdown 的「伪流」保留当前 50-char/10ms 回放，但**把 10ms 加大到 20-30ms 并让 chunk 切到 200-400 字符**，否则 10 个角色同时回放 = 每秒 10×100 = 1000 条 SSE event，前端 re-render 帧率会崩。
  3. UI 层在 `role_started` 后立即渲染「角色卡片（占位 skeleton + 活动 spinner）」，不要等 `role_done` 才出现卡片。v0.1 §5.4 没强调这点，必须补进 MVP checklist。

## Major（MVP 可以先过但 M2 前必改）

### 1. **`/api/v1/collaboration/roles` 元数据来源含糊，建议"静态 yaml 为主，hermes list 为校验"**
- **引用**：v0.1 §4.2.1「元数据从静态 yaml 或从 agent 服务拉」、§八「Skill 列表解析脆弱」。
- **问题**：`hermes_cli.py:28-47 list_skills()` 靠正则 `│` 字符解析 hermes CLI 的表格，hermes 升级格式即炸（v0.1 §八已承认）。前端角色勾选面板每次拉活运行时列表 = 运行时 UI 爆炸。
- **推荐方案**：
  - 新增 `agent/configs/gstack_roles.yaml`（23 条，slug + 中文职能 + 一句话能力 + 分类 + 推荐场景，字段与 `gstack-roles.md` 表格对齐）。
  - `GET /api/v1/collaboration/roles` 只读这个 yaml（无 CLI 调用）→ UI 永不炸。
  - 启动时另跑一次 `hermes skills list` 做「yaml 声明的 23 个是否全部实装」的 health check，失败记 warning（不 block 启动）。
  - 未来扩到 24/25 人 = 改 yaml + 升级 gstack 版本，两件事解耦。

### 2. **DB schema 缺少多个必填字段**
- **引用**：v0.1 §4.2.3。
- **缺项**（按重要度）：
  1. `collaboration_sessions.parent_conversation_id INTEGER NULL` — 否则无法从某个普通对话「升级为协作」或在会话列表里关联，未来 shared 链接需求也要它。
  2. `collaboration_sessions.cancel_reason TEXT NULL` + `cancelled_at TIMESTAMP NULL` — 用户中途关掉页面/主动取消时要记录；不然 `status=failed` 里面鱼龙混杂无法归因。
  3. `role_reports.started_at / finished_at TIMESTAMP` — 不要只存 `duration_ms`，时间线视图要画甘特图；`duration_ms` 对同时开跑的角色无法还原「谁卡住了」。
  4. `role_reports.cost_input_tokens / cost_output_tokens / cost_usd REAL` — 10 角色协作单次成本 0.5-3 USD，必须按角色归因，未来做配额/billing 需要。MVP 就埋上，比后面迁移便宜。
  5. `role_reports.error_type TEXT NULL`（`timeout / rate_limit / skill_missing / script_blocked / exception`）区别于 freetext `error`。
  6. `collaboration_sessions.model / provider TEXT` — §4.2 §4.3 都丢了 provider（bridge `ChatRequest` 里的字段全没传，见 hermes-integration-map §5），导致同一次会话可能混用多家模型且无法复现。
- **最小修正**：migration `0003_collaboration.sql` 一次性加好上面 6 列，字段加了空着不痛，漏了补迁移表结构痛。

### 3. **部分角色失败时的汇总摘要需要明确策略，不能只"标红"**
- **引用**：v0.1 §七「单个角色失败 → 其他角色继续，最终 summary 显示失败项」。
- **问题**：10 角色里 3 个失败，coordinator LLM 如果拿到 7 份成功报告 + 3 条 `failed: timeout/rate_limit`，需要明确 prompt 策略：
  1. 汇总摘要**仅基于成功的 N-K 份** —— 给 coordinator 明确指令「仅对下列成功报告归纳，失败角色单独列出，不要试图脑补」。
  2. 失败项要区分**可重试**（429/timeout）与**不可重试**（skill 未装/script blocked），UI 上给不同 CTA。
  3. 若 `K > N/2`（半数以上失败）—— 摘要节点应直接返回「协作质量不足，建议重跑」，不要硬挤一份垃圾摘要。
  4. 汇总 LLM 自身也可能失败（它也是 LLM API 调用），要有降级：`coordinator_failed → 前端直接把 N 份报告拼接展示，不做摘要`。
- **修正**：§七「错误处理」单独加一节「汇总策略」，且 §4.1.5 SSE 事件补 `coordinator_started / coordinator_skipped{reason}`。

### 4. **`MAX_CONCURRENT_WORKERS=5` 是模块级全局常量，调高会污染普通对话**
- **引用**：v0.1 §八「worker_orchestrator 的 5 并发限制...调高会同时影响普通对话性能，需谨慎（建议新节点独立参数）」；代码 `agent/app/graph/nodes/worker_orchestrator.py:11 MAX_CONCURRENT_WORKERS = 5`。
- **问题**：它是**模块级常量**而非类/graph 实例字段（见上面代码行）。两条路径共用同一个 Python 进程时，semaphore 是 per-call 新建的（`worker_orchestrator_node:47 asyncio.Semaphore(MAX_CONCURRENT_WORKERS)`），所以**不是真的"全局共用一个 semaphore"**——但如果 v0.1 §七直接把这个常量改到 10，普通对话的 planner 产出 10 个 step 也会并发 10 个 Worker，每个 Worker 一次 LLM 调用 → 单对话突然变 10 个 Claude 并发，碰 429。
- **修正**：
  - 不要动 `MAX_CONCURRENT_WORKERS`。
  - 在 `CollaborationOrchestrator` 独立定义 `COLLAB_MAX_CONCURRENT`，默认 3，从 `settings.COLLAB_MAX_CONCURRENT` 读，env 可覆盖。
  - v0.1 §七「MAX_CONCURRENT_WORKERS=5 不够」这句删掉，替换成「新增 `COLLAB_MAX_CONCURRENT=3`，与 `MAX_CONCURRENT_WORKERS` 解耦」。

### 5. **预装脚本的"幂等"定义不够，会丢用户本地改动 / 破坏版本锁定**
- **引用**：v0.1 §4.3.1「**幂等**：已存在的 skill 先 rm 再 copy」。
- **问题**：
  - 「先 rm 再 copy」不是幂等，是「强制覆盖」。如果运维临时在服务器改了某个 SKILL.md（调 prompt、加示例），重跑预装脚本会**静默抹掉**。
  - gstack 版本锁定未定（§八也承认）。现在 `hermes_cli.py:117-154 install_hub_skill` 直接 `hermes skills install <identifier>`，等于跟踪 hub latest，可复现性为 0。下次部署可能拿到和上次不同的 SKILL.md。
- **推荐方案**（写进 §4.3.1）：
  1. **gstack 用 git submodule 锁定 commit sha**：`git submodule add https://.../gstack vendor/gstack`，预装脚本直接从 `vendor/gstack/{role}/` copy，不走 hermes hub。hermes hub 只在"用户自助安装其它 skill"时走。
  2. 预装脚本打 hash 校验：目标目录已存在且 `SKILL.md` 的 sha256 与源一致 → skip；不一致 → 备份到 `skills.bak.<timestamp>/` 再覆盖。
  3. 版本信息落盘 `agent/skills/.gstack_version`（存 submodule commit sha），`/api/v1/collaboration/roles` 响应里带上，前端 footer 可显示「gstack @ abc1234」。

### 6. **超时策略串联有 bug：`整个会话超时 = max(单角色) + 30s` 对并发场景不成立**
- **引用**：v0.1 §七「单角色 hermes CLI 执行 ≤ 300s；整个协作会话超时 = max(单角色 timeout) + 30s」。
- **问题**：10 个角色并行时「max(单角色 timeout)」就是 300s，整个会话超时 330s。但若 RoleWorker 排队（concurrent=3，10 个角色分 4 批），最差 `ceil(10/3)*300 = 1200s`，会话层 330s 超时会在大部分角色还没启动时就爆。
- **修正**：会话超时应 ≈ `(ceil(N / concurrent) * single_timeout) + coordinator_time + margin`。取具体数：MVP concurrent=3, N<=10, single=300s → 会话 soft cap ~1200s，再加 60s coordinator + 60s margin = **1320s（22 分钟）**。前端要有「会话>5min 仍在跑」的明显提示，不要看起来像卡死。
- 同时：单角色 timeout 300s 对 `investigate` 类深度调查任务可能不够，建议 §七加一句「允许 per-role override」。

### 7. **hermes CLI stderr 完全丢失，错误归因困难**
- **引用**：`hermes_cli.py:20-25` `subprocess.run(stdout=PIPE, stderr=PIPE)` 把 stderr 读了但 `execute_skill` 返回的 `result.stdout.strip()` **丢弃了 stderr**。
- **问题**：hermes 失败信息（缺 API key、skill 未装、API 429）都写在 stderr。现在 bridge 返回 `{"result": ""}` 给 agent，agent 误以为 skill 返回空字符串成功，`role_reports.status=done, markdown=""`。用户看到的是空卡片而非「⚠ 失败原因：OPENAI_API_KEY not set」。
- **修正**（Major 但在 M1 做修改量很小）：
  - 在 `hermes_cli.execute_skill` 里判断 `result.returncode != 0 or not result.stdout.strip()`，把 stderr 一起抛到上层。
  - bridge `execute_skill` 返回格式加 `exit_code / stderr` 字段。
  - agent 侧 RoleWorker 看到这些字段才能正确设置 `role_reports.status=failed, error_type=...`。
  - 否则 §七「Hermes CLI not found / skill 未装 → 提前检查，报明确错误」就是空话。

### 8. **鉴权/隔离：`collaboration_sessions.user_id` 足够做当前用户隔离，但共享链接需求会破坏 JWT 模型**
- **引用**：v0.1 §七「用户只能查自己的会话」、§八 Q5「协作会话的分享链接需求是否在 MVP 范围内」。
- **问题**：
  - 当前模型 = `AuthMiddleware + user_id 过滤`，前端凭 JWT 拉 `GET /api/v1/collaboration/sessions/:id`，隔离 OK。
  - 一旦 Q5 落地「分享只读链接」，单纯 user_id 隔离不够，要么加 `share_token TEXT UNIQUE`（URL 内嵌）、要么加 `visibility ENUM(private/org/public)`。M1 DB schema 埋 `share_token` 空字段（预留），代价 0；M2 补 handler。不埋 → 后面改生产表。
  - 还有一个遗漏：`role_reports.artifacts` 里若含文件下载链接（v0.1 §4.1.1 提到 artifacts），文件 URL 的 access control 要独立做，不能只靠 session 的 user_id（因为 artifact 的 URL 往往被前端缓存/分享到 IM）。MVP 至少要求 artifact URL 携带 HMAC 签名 + 过期时间。
- **修正**：
  1. DB schema 加 `share_token TEXT UNIQUE NULL`、`visibility TEXT DEFAULT 'private'`。
  2. `role_reports.artifacts` 的 JSON 结构显式约定 `{"name", "url", "mime", "size", "expires_at"}`；不用明文路径。
  3. §七「鉴权」加一条「artifact URL 用签名链接，24h 过期」。

## Nit（锦上添花）

### 1. 场景 yaml 路径要定死
v0.1 §4.1.4 原话「放在 agent 服务目录 `agent/configs/` 或 repo 顶层，二选一」—— 评审通过前就得选定。推荐 `agent/configs/tool_scenarios.yaml`（贴近 python 代码，部署跟 agent 包一起走）。别留「二选一」到开发阶段决定，否则 import path 来回改。

### 2. §4.1.5 SSE 事件命名与现有不一致
现有事件是 `worker_started / worker_progress / worker_done`（见 `worker_orchestrator.py:40-103`），v0.1 新增的是 `role_started / role_progress / role_done / collaboration_done`。建议前端 store 用一个统一的 union type，或在 `collaboration_done` 之外再加 `coordinator_started / coordinator_done`，避免 UI 进度条缺最后一段。

### 3. `RoleReport` 字段命名 `markdown` 太窄
gstack 里 `design-shotgun` 会产多张图片、`canary` 会抓 console log、`qa` 会带测试 matrix。建议 `content` 字段类型改 `{"type": "markdown" | "html" | "mixed", "body": "...", "blocks": [...]}`，否则到 M2 又迁移。

### 4. 导出 Markdown 的文件名约定缺失
§6 M1 有「Markdown 导出」但没说文件名 / 目录结构 / 角色顺序。建议：`collab-<session_id>-<yyyymmdd>.md`，章节顺序按角色在 `selectedRoles` 里的原始顺序，不要按完成时间（完成时间受调度影响不稳定）。

### 5. 前端勾选超过推荐角色数的 UX 未定
UI 要加"已勾选 10/23（推荐 3-5 个）"的引导文案+颜色变化，避免用户一口气勾 20 个再抱怨慢。

## 赞同的设计（避免被后续迭代推翻）

### 1. **MVP 只做"并行 + 汇总"，串行 handoff / supervisor 延后**
v0.1 §三 → §六 这条切分正确。串行 handoff 需要定义角色间数据契约（前者的 `RoleReport` 如何变成后者的 input），这是 M3 的独立大块，和当前 23 角色对接是两件事。不要用 MVP 时间把二者混做。

### 2. **`selectedSkill`（单选）和 `selectedRoles`（多选）两条路径并存**
v0.1 §八「两条路径并存，别互相污染」完全正确。保留现有单 skill 对话不动（`selectedSkill` 在 chat 页），协作只在新页面生效。试图统一会动现有 ChatPanel，风险高收益低。

### 3. **工具场景放后端静态 yaml**
v0.1 §三锁定。理由在于：MVP 用户不需要自定义场景（Q5 调研都没做），运维改 yaml 发版是可接受节奏；前端编辑工具集合引入的复杂度（校验、版本化、权限）是 M3 级别的工作。决策正确。

### 4. **DB 独立 `collaboration_sessions` 表**
不混进 `conversations` 表是对的。协作会话的生命周期、权限、导出需求都和普通对话不同，复用一张表会把 status/role/artifact 字段塞进去污染原表。独立表 + `parent_conversation_id`（见 Major #2）关联即可。

## 额外风险（文档未提但你发现的）

### 1. **bridge `/api/v2/skills/{name}/execute` 完全没带上下文**
`chat_handler.py:112-121 execute_skill` 调 `hermes_cli.execute_skill(name, req.message)`，只传 `message`。`req.session_id / user_id / conversation_id / model / provider` 全丢（`hermes-integration-map.md §5` 也点名了）。意味着：
- hermes 跑 skill 时看不到同一协作会话里其他角色的产出（未来 M2 的 shared scratchpad 需求直接死）。
- 用户选了 provider=qwen / model=sonnet，hermes 仍用自己的默认 Claude，**billing 和日志归因混乱**。
MVP 至少把 `model / provider` 透传下去（hermes CLI 支持 `--model`），否则 v0.1 §4.2 的 provider 选择等于摆设。

### 2. **RunSkillScriptTool 的 AST 安全检查对 hermes 预装的 skill 会全量 block**
`skill_tools.py:14-15 BLOCKED_MODULES = {"os", "subprocess", ...}` + `BLOCKED_BUILTINS = {"open", ...}`。gstack skill 自带的 python helper script（如果有）肯定会 `import os / open()`，全部会被 blocked。
- 这和 Blocker #1 是同一个坑的两面：MVP 决定**不要**尝试用 RunSkillScriptTool 跑 gstack skill 的 Python 脚本，全部走 hermes CLI 路径。
- 否则要为 gstack skill 单独放宽沙箱，会打开大面积安全漏洞。

### 3. **没有取消机制**
v0.1 §六 M2「会话历史列表与复用」，但从头到尾**没提中途取消**。用户点「开始协作」后发现选错角色，当前路径：只能关闭浏览器（后端继续跑 5-20 分钟），浪费 LLM token。MVP 应加：
- `POST /api/v1/collaboration/sessions/:id/cancel` → 设置 `cancel_reason`，orchestrator 侧用 `asyncio.CancelledError` 终止 gather。
- 注意 `subprocess.run` 是阻塞的，即使 Python 层 cancel 了，bridge 那边的 hermes 进程不会被杀（leak 风险）；要改用 `asyncio.create_subprocess_exec` 或 bridge 加进程管理。

### 4. **没有 retry / rate-limit 退让策略**
Claude/OpenAI 429 在 10 并发下必现。现在 Worker._execute_tool 直接把 429 当失败返回。应：
- RoleWorker 看到 `error includes "429" or "rate_limit"` → 退避 30-60s 后重试一次（总共 2 次），超过还失败 → mark failed。
- 重试事件也要 emit SSE，前端显示 "rate-limited, retrying in 30s"，否则用户看到的还是空白。

### 5. **observability 缺失**
v0.1 §七「Gateway 侧打 metric」但 agent 侧没提 tracing。10 角色并行时哪个慢、哪个重试、哪个 429 —— 仅靠日志事后查根因极痛。建议 MVP 就把 OpenTelemetry span 埋进 `CollaborationOrchestrator` + `RoleWorker` + `HermesRoleTool`（span attributes: session_id / role_slug / attempt / duration）。后续接 Jaeger/Tempo 是 1 天工作量，不埋就永远补不上。

### 6. **gstack 升级 = 全量覆盖 23 个角色 = 运行中的会话产出 SKILL.md 被中途换掉**
预装脚本跑一次，M1 用户侧会话进行中，如果运维正好触发 hot reload（`agent/api/v1/skills/admin/reload`）—— 内存里的 skill manifest 变了但子进程里已经拷走的旧 prompt 还在。目前的表现是「早开始的会话用旧 prompt，晚开始的用新 prompt」，日志里会出现同一天同一角色两个版本输出。
修正：`collaboration_sessions` 建议加 `gstack_version TEXT`（commit sha），落库时快照一次；复现 bug 时能对齐版本。
